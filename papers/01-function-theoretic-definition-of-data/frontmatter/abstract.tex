\begin{abstract}
\emph{Data} is one of the most frequently used and least precisely defined
words in computing. The references that do define it converge on
something like ``raw facts,'' which is not wrong, but it never specifies
the structure a collection of facts actually needs before a machine can
index it, compare it, or move it across a wire. That gap does not stay
abstract for long: it resurfaces as schema drift, as serialization
defects, and as a whole class of type-confusion vulnerabilities, all
arising from the same root cause---a fixed byte sequence quietly
reinterpreted under an indexing or typing assumption nobody wrote down.
This paper argues for treating data formally as a \emph{function from an
index set to a value set}, states that definition with explicit
well-definedness conditions, and places it against three existing
frameworks: Floridi's General Definition of Information, Zins' faceted
classification, and the DIKW hierarchy. From the definition we derive four
corollaries connecting it directly to recurring failure modes in software
engineering---implicit schema coupling, byte-level reinterpretation,
unsound equality checks, and ambiguous serialization contracts---each with
a worked example.
\end{abstract}

\paragraph{Keywords} data representation; formal definition; information theory;
type safety; serialization; systems design; data semantics.
